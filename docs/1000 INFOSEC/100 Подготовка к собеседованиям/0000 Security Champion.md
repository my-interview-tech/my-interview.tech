_Безопасная разработка (**SSDLC** (Secure Software Development Lifecycle)) – это процесс создания программного обеспечения с учетом аспектов информационной безопасности с самого начала жизненного цикла разработки до завершения эксплуатации. Этот подход интегрирует методы, практики и инструменты для обеспечения защиты от угроз, выявления и устранения уязвимостей, что позволяет предотвратить атаки и сохранить конфиденциальность, целостность и доступность данных._

Набор установленных методов и ролей, которые предусматривают распределение обязанностей по безопасности между различными сотрудниками в организации. 

**Главная цель** - повышение общего уровня осведомленности о безопасности и интеграция ее в процессы компании.

**Security Champion** является заинтересованным в области безопасности специалистом, он помогает наладить процессы между отделом разработки и ИБ.

**Кто им может быть?**

Разработчик, тестировщик, аналитик, девопс, архитектор - практически любой инженер.

**Требования к кандидату**

1. Искреннее желание развиваться в ИБ
2. Понимание что придется много учиться (даже в нерабочее время)
3. Понимание функционального взаимодействия между командами
4. Уверенность в себе
5. Желание сделать продукты компании безопаснее

**Выполняемые функции (с градацией)**

В зависимости от готовности тратить определенное время, будут добавляться новые функции:

|                                      5-10%                                       |                     10-15%                      |                                                                     15-20%                                                                     |
| :------------------------------------------------------------------------------: | :---------------------------------------------: | :--------------------------------------------------------------------------------------------------------------------------------------------: |
| Анализирует бизнес-логику продукта или собирает информацию, если её недостаточно | Анализирует уязвимости и предлагает исправления | Готовит рекомендации для команды по безопасной разработке по стеку, основываясь на найденных в продукте уязвимостях (или по своему пониманию). |
|    Актуализирует/создает таблицу с API и репозиториями продукта (при наличии)    | Делегирует устранение уязвимостей в разработку. |                                                    Участвует в ревью кода и в тестировании.                                                    |
|              Вовлекается во взаимодействие отделов разработки и ИБ               |                                                 |                                                                                                                                                |

**Процесс внедрения**

1. Определить роль Security Champion в команде (путем добровольного волеизъявления)
2. Направить сотруднику рекомендации по обучению, ввести его в курс дела
3. Формально описать процессы и процедуры взаимодействия, которые определяют как именно будет работать и уведомлять сотрудник отдел ИБ
4. Направить всем остальным сотрудникам рекомендации по обучению, чтобы повысить осведомленность
5. Security Champion регулярно следит за последними тенденциями в мире ИБ, следуя рекомендациям архитектора ИБ, DevSecOps, СТО

**Процессы и процедуры взаимодействия между разработкой и отделом ИБ**

1. Security Champion получает список уязвимостей от DevSecOps/AppSec инженера по мере их верификации (еженедельно/ежемесячно)
2. В случае обнаружения проблем уровня CRITICAL и HIGH необходимо самостоятельно проанализировать проблему, попытаться ее сразу же исправить с командой разработки
3. Если исправление было успешными, уведомить ИБ отдел об успешном исправлении (и/или DevSecOps инженера)
4. Если компетенций у Security Champion не хватило, то проблема выносится на уровень выше, и в первую очередь начинается обсуждение с отделом ИБ
5. Оценивается риск уязвимости, ее опасность (желательно посчитать CVSS в рамках проекта)
6. В исправлении уязвимости принимают участие выделенные ресурсы команды и отдел ИБ
7. В случае успешного устранения командой разработки, уведомить ИБ отдел (и/или DevSecOps инженера)

**Задачи:**

* Анализ результатов SAST, SCA, TRIVY
* Поиск уязвимостей, IDOR и проблемы бизнес-логики
* Определение потенциально небезопасных конструкций в коде
* Поддержка команды разработки в вопросах безопасности

**Примеры хорошего Security Champion**

1. Умеет работать в команде
2. Ответственно подходит к решению задач
3. Может довести дело до конца
4. Заинтересован в изучении ИБ
5. Имеет обширные знания в программировании, информационной безопасности, инфраструктуре
6. Понимает важность информационной безопасности в современном мире
7. Постоянно дает полезные советы внутри команды разработки, повышает общий уровень вовлеченности в развитие информационной безопасности и цифровой гигиены

**Базы знаний об уязвимостях:**

1. [Common Vulnerabilities and Exposures](https://www.cve.org/) (CVE) - база данных общеизвестных уязвимостей информационной безопасности. Каждой уязвимости присваивается идентификационный номер вида CVE-год-номер, описание и ряд общедоступных ссылок с описанием.
2. [Common Weakness Enumeration](https://cwe.mitre.org/index.html) (CWE) - общий перечень дефектов (недостатков) безопасности.
